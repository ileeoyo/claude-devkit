# Git Commit 助手

分析当前Git项目的代码改动，生成规范的 commit message，经用户确认后执行提交。

## Guardrails
- 如果项目有指定的 commit 规范，遵循项目规范；否则使用通用格式
- commit message 格式：`类型: 描述`，**禁止添加 scope、模块标识等任何额外成分**（如 `fix(api-doc)`、`feat(module):` 都是错误的）
- 禁止添加 `Co-Authored-By` 或 `Generated with Claude Code` 后缀
- 不要自动 push 到远程仓库，除非用户明确要求
- 默认以本次所有变更文件（**包括未追踪的新文件**）作为提交候选范围，**不要依赖用户当前的暂存状态**
- 如果用户指定忽略或排除某些文件，则这些文件必须同时从 commit message 分析范围和最终提交范围中剔除
- 不要使用 `git add -A` 或 `git add .`，应先明确本次提交的文件清单与排除清单，再按文件列表执行暂存或取消暂存

## 敏感文件检测与处理
- 提交前检查是否包含敏感文件（.env、credentials、*.pem、*.key、password、secret 等）
- 如果检测到敏感文件，立即警告用户并列出敏感文件清单
- 用户必须选择以下操作之一：
  1. **取消提交**：终止本次提交操作
  2. **忽略敏感文件**：从提交中排除敏感文件，仅提交非敏感文件
  3. **确认提交**：用户二次确认，知悉风险并执意提交（需用户明确输入"确认"）

## Steps
1. 先确定本次提交的候选文件范围：默认以当前仓库内所有有改动的文件为候选范围，不区分用户原本是否已暂存；如有未跟踪的新文件需要提交，也一并纳入候选范围
2. 运行 `git status && git diff HEAD` 一次性查看工作区状态和所有改动内容（包含已暂存和未暂存的差异）；对于未追踪的新文件，必须额外读取文件内容并纳入分析范围，确保分析范围覆盖本次候选文件
3. 检查候选范围内是否存在敏感文件，如有则按上述敏感文件处理流程执行
4. 基于“最终准备提交的文件范围”分析改动，确定：
   - **类型**：feat（新功能）、fix（修复）、refactor（重构）、docs（文档）、style（格式）、test（测试）、chore（杂项）
   - **描述**：简洁描述改动内容（中文）
5. 生成 commit message，格式如下：

   ```
   类型: 总结性描述

   - 改动点1
   - 改动点2
   - 改动点3
   ...
   ```

   **总结性描述规则**应简洁精炼，一行内完成。

   **分项改动点描述规则**：
   - 分项描述主要的改动点，最多不超过 **5个**，避免大改动时分项过多
   - 每个分项应简洁明了，说明"做了什么"而非"怎么做的"
   - 分项之间避免重复，聚焦不同方面的改动
   - **分项排序**：按重要程度、影响程度、影响范围、功能大小降序排列，越重要、影响越大、范围越广、功能越复杂的改动点排在越前面
6. 将 commit message 展示给用户，询问确认。**每次提交必须经过用户授权，未经确认不得执行提交**
   - 用户可能修改 message 或提出修改意见（比如忽略某些文件）
   - 如果用户指定忽略或排除某些文件，则这些文件即使原本已经暂存，也必须从分析范围中剔除，并在提交前从暂存区移除
   - 调整排除范围后，需要基于更新后的最终提交文件范围重新生成并确认 commit message
   - **循环此步骤直到用户明确确认**，或用户选择取消提交
7. 用户确认后，先将最终提交范围内的文件全部加入暂存，再将排除文件从暂存区移除，最后执行提交；必须确保最终暂存内容与已确认的 commit message 对应

## Commit 类型说明

| 类型 | 说明 | 示例 |
|-----|------|------|
| feat | 新增功能 | `feat: 新增用户注册功能` |
| fix | 修复 bug | `fix: 修复登录验证重复问题` |
| refactor | 重构代码 | `refactor: 重构工具类方法` |
| docs | 文档变更 | `docs: 更新部署说明` |
| style | 代码格式 | `style: 格式化代码` |
| test | 测试相关 | `test: 新增用户服务单元测试` |
| chore | 构建/工具 | `chore: 升级依赖版本` |
